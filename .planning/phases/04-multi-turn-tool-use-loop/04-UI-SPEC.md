## Layout & Structure

### Proposal Bubble Layout

```
┌──────────────────────────────────────────────────────┐
│ [🤖 avatar] [Bot bubble — left-aligned, muted bg]     │
│              [proposal text in markdown]              │
│              [  Confirm  ]  ← inline, coral           │
└──────────────────────────────────────────────────────┘
```

**Anatomy:**
- Avatar: same coral `Bot` icon as normal bot bubble
- Bubble: same muted background as normal bot bubble (`bg-muted`)
- Proposal text: rendered with `ReactMarkdown` + `remarkGfm` (bold, lists, tables all supported — e.g. "**Action:** Create contract for [name], 10 sessions PT, 1,500,000 VND?")
- Confirm button: inline inside bubble, `mt-2` top margin, coral background, white text, same height as send button (`h-8`)

### Proposal Bubble — After User Confirms

```
┌──────────────────────────────────────────────────────┐
│ [🤖 avatar] [Bot bubble — muted]                     │
│              "✓ Confirmed — creating contract..."     │
│ [ user bubble — italic] "Confirmed"                   │
│ [🤖 avatar] [Bot bubble — result card]               │
└──────────────────────────────────────────────────────┘
```

**"Confirmed" user bubble:** User message bubble with italic `text-sm italic text-muted-foreground` text (no coral background). Visually distinct from a normal user message — communicates "system-level confirmation" rather than free-text user input. Does NOT use `react-markdown`.

## Interaction Contracts

### Complete Two-Call Flow

```
User types "tạo hợp đồng mới" + presses Send
    ↓
MessageInput: addMessage({ role: 'user', content: "tạo hợp đồng mới" })
setLoading(true)
POST /api/ai-chatbot
    ↓
┌─ API returns type === 'proposal' ──────────────────────┐
│ addMessage({ role: 'assistant', content: proposalText,  │
│              type: 'proposal' })                        │
│ setLoading(false)                                       │
│ [Confirm button visible in bubble]                      │
│                                                          │
│ User clicks Confirm                                      │
│    ↓                                                    │
│ addMessage({ role: 'user', content: "CONFIRMED: create_contract", │
│              type: 'proposal' })  ← italic bubble       │
│ setLoading(true)                                        │
│ POST /api/ai-chatbot  (2nd call, with CONFIRMED msg)    │
│    ↓                                                    │
│ addMessage({ role: 'assistant', content: resultText }) │
│ setLoading(false)                                       │
└─────────────────────────────────────────────────────────┘
┌─ API returns type === 'text' (no confirmation needed) ─┐
│ addMessage({ role: 'assistant', content: data.reply })  │
│ setLoading(false)                                        │
└─────────────────────────────────────────────────────────┘
```

### Proposal Detection in `MessageInput.tsx`

```typescript
// After fetch, where data = await res.json()
if (!res.ok || data.type === 'error') {
  addMessage({ role: 'assistant', content: data.error ?? 'Something went wrong. Please try again.', type: 'error' })
} else if (data.type === 'proposal') {
  // Show proposal bubble (type === 'proposal' carries through)
  addMessage({ role: 'assistant', content: data.reply, type: 'proposal' })
  setLoading(false)
} else {
  addMessage({ role: 'assistant', content: data.reply })
}
```

**Key:** `setLoading(false)` is called inside the `type === 'proposal'` branch — the loading indicator disappears and the confirm button becomes active immediately.

### Confirm Button Handler

The `onConfirm` callback is passed through `MessageList` → `MessageBubble` and wired in `MessageInput.tsx` via a `ref` or closure pattern. The exact mechanism is in the executor's discretion (can use a `Map<id, onConfirm>` or a Zustand action for the pending proposal).

**Minimum viable approach:** Use a Zustand `pendingConfirm` state or a module-level variable in `MessageInput.tsx` to store the `onConfirm` resolver between render cycles.

## API Response Contract (Phase 4 update)

### Success — Text (read-only action or final result)

```json
{
  "reply": "Here's your contract list...",
  "type": "text",
  "role": "CUSTOMER"
}
```

### Success — Proposal (write action requires confirmation)

```json
{
  "reply": "**Action: Create contract**\n\nCustomer: [name]\nPlan: 10 sessions PT\nPrice: 1,500,000 VND\n\nReady to proceed?",
  "type": "proposal",
  "role": "ADMIN"
}
```

### Success — Error

```json
{
  "reply": "Permission denied — you need ADMIN role to delete contracts.",
  "type": "error",
  "role": "STAFF"
}
```

The `type === 'proposal'` discriminator tells the client to render the confirm button. The executor decides the `CONFIRMED:` message content format (e.g., `CONFIRMED: create_contract` or `CONFIRMED: create_contract__customer=123__amount=1500000`). The route returns `type: 'text'` for the second call (final result after confirmation).
