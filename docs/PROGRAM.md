## 1. Application Overview

GoChul Fitness is a **gym/fitness studio management application** for managing:
- **Contracts** between customers and personal trainers (PT) / rehab providers
- **Training sessions (history)** scheduled within those contracts
- **User onboarding** via an essential information questionnaire
- **Role-based access** for ADMIN, STAFF, and CUSTOMER

**Core business flow:**
1. STAFF/ADMIN creates a contract for a customer (status = `NEWLY_CREATED`)
2. Customer reviews → confirms → pays → PT confirms → customer activates contract
3. Once ACTIVE, sessions (history) are booked against the contract
4. Sessions are checked-in by both customer and staff
5. Contracts/sessions expire based on date and time rules

## 2. Database Structure (InstantDB)

InstantDB is used as the primary database. The schema is defined in `instant.schema.ts` and permissions in `instant.perms.ts`.

### 2.1 Entities

#### `$users` (built-in InstantDB entity)
| Attribute | Type | Notes |
|-----------|------|-------|
| `id` | string | Auto-generated by InstantDB |
| `email` | string? | Optional, unique, indexed |

#### `contract`
| Attribute | Type | Notes |
|-----------|------|-------|
| `id` | string | Auto-generated by InstantDB |
| `created_at` | number | Unix timestamp (ms) |
| `updated_at` | number | Unix timestamp (ms) |
| `start_date` | number? | Contract start date (timestamp) |
| `end_date` | number? | Contract end date (timestamp) |
| `kind` | string | `PT`, `REHAB`, or `PT_MONTHLY` |
| `credits` | number? | Number of sessions for PT/REHAB contracts |

| `purchased_by` | string | InstantDB `$users.id` of the customer (link) |

**Contract Status values (`ContractStatus`):**
```
NEWLY_CREATED → CUSTOMER_REVIEW → CUSTOMER_CONFIRMED → CUSTOMER_PAID → PT_CONFIRMED → ACTIVE
                ↓                  ↓                   ↓                ↓
              CANCELED           CANCELED             CANCELED         CANCELED
                                                    (by ADMIN only)   (by ADMIN only)
```
Terminal statuses: `CANCELED`, `EXPIRED`

**Contract Kind values (`ContractKind`):**
- `PT` — Personal Training (credits-based)
- `REHAB` — Rehabilitation (credits-based)
- `PT_MONTHLY` — Monthly PT (not credits-based)

#### `history` (training sessions)

| `created_at` | number | Unix timestamp (ms) |

| `status` | string | See HistoryStatus below |
| `from` | number | Start minute of session (e.g., 480 = 8:00 AM) |
| `to` | number | End minute of session (e.g., 570 = 9:30 AM) |
| `teach_by` | string | InstantDB `$users.id` of trainer (link) |
| `staff_note` | string? | Note written by STAFF/ADMIN |
| `customer_note` | string? | Note written by CUSTOMER |
| `user_check_in_time` | number? | Timestamp when customer checked in |
| `staff_check_in_time` | number? | Timestamp when staff checked in |

**History Status values (`HistoryStatus`):**
```
NEWLY_CREATED → CHECKED_IN
      ↓
   CANCELED       (NEWLY_CREATED only, when session time passes → EXPIRED)
      ↑
   EXPIRED
```

### 2.3 InstantDB Permissions (`instant.perms.ts`)

All permissions are currently set to allow reads for all authenticated users. Write permissions are controlled by API-level role checks (not InstantDB permissions).

```typescript
contract:   allow view: true,  create/delete/update: false
history:    allow view: true,  create/delete/update: false
user_setting: allow view: true, create/delete/update: false
```

> **Important:** Actual permission enforcement is done in the API routes, not in InstantDB permissions. InstantDB permissions here serve as a secondary layer.

|------|---------|
| ADMIN | ✅ |
| STAFF | ✅ |
| CUSTOMER | ❌ (403 Forbidden) |

New contracts are always created with status `NEWLY_CREATED` and `sale_by` = the authenticated user's InstantDB ID.

#### Updating Contract Fields (`/api/contract/update`)
| Role | Allowed |
|------|---------|
| ADMIN | ✅ All fields |

- **ADMIN**: Soft-deletes by setting status to `CANCELED`

#### Updating Contract Status (`/api/contract/updateStatus`)

| Transition | ADMIN | STAFF | CUSTOMER |
|------------|-------|-------|---------|
| `NEWLY_CREATED` → `CUSTOMER_REVIEW` | ✅ | ✅ | ❌ |
| `NEWLY_CREATED` → `CANCELED` | ✅ | ❌ | ❌ |
| `CUSTOMER_REVIEW` → `CUSTOMER_CONFIRMED` | ✅ | ❌ | ✅ (own contract) |
| `CUSTOMER_REVIEW` → `CANCELED` | ✅ | ❌ | ✅ (own contract) |
| `CUSTOMER_CONFIRMED` → `CUSTOMER_PAID` | ✅ | ❌ | ✅ (own contract) |
| `CUSTOMER_CONFIRMED` → `CANCELED` | ✅ | ❌ | ✅ (own contract) |
| `CUSTOMER_PAID` → `PT_CONFIRMED` | ✅ | ✅ | ❌ |
| `CUSTOMER_PAID` → `CANCELED` | ✅ | ❌ | ❌ |
| `PT_CONFIRMED` → `ACTIVE` | ✅ | ❌ | ✅ (own contract) |
| `PT_CONFIRMED` → `CANCELED` | ✅ | ❌ | ❌ |
| `ACTIVE` → `CANCELED` | ✅ (force) | ❌ | ❌ |
| Any → `EXPIRED` | ✅ (force) | ❌ | ❌ |

**Note:** `ACTIVE` contracts cannot be canceled by STAFF or CUSTOMER — only ADMIN can force-cancel an active contract.

#### Getting Contracts (`/api/contract/getAll`)
| Role | Scope |
|------|-------|
| ADMIN | All contracts |
| STAFF | Contracts where `sale_by` = their user ID |
| CUSTOMER | Contracts where `purchased_by` = their user ID; `NEWLY_CREATED` is always excluded |

### 4.3 History / Session Permissions Detail

#### Creating Sessions (`/api/history/create`)
| Role | Constraint |
|------|-----------|
| ADMIN | No constraints |
| STAFF | No constraints (can create for any contract) |
| CUSTOMER | Can only create for contracts they `purchased_by` |

Additional checks:
- Contract must be `ACTIVE`
- Contract must not be expired (`end_date` not in the past)
- Session time must not overlap with other sessions by the same trainer
- Duration must match `duration_per_session` from the contract
- Credits check: for PT/REHAB contracts, `used_credits < total_credits`

#### Updating Sessions (`/api/history/update`)
| Role | Constraint |
|------|-----------|
| ADMIN | Any session |
| STAFF | Only sessions they created (`users` link) |
| CUSTOMER | Only sessions from contracts they `purchased_by` |

#### Deleting Sessions (`/api/history/delete`)
| Role | Constraint |
|------|-----------|
| ADMIN | Any session |
| STAFF | Only sessions they created |
| CUSTOMER | Only sessions from contracts they `purchased_by` |

#### Check-in / Status Update (`/api/history/updateStatus`)

**Cancel (`CANCELED`):** Only `NEWLY_CREATED` sessions can be canceled.
**Check-in (`CHECKED_IN`):** Dual-check-in system:
- CUSTOMER can set `user_check_in_time`
- STAFF/ADMIN can set `staff_check_in_time`
- When **both** timestamps are set → status becomes `CHECKED_IN`
- If one is already set, the other can still update the status to `CHECKED_IN`
- Each role can only check in once (idempotent)

#### Session Notes (`/api/history/updateNote`)
| Role | Field | Constraint |
|------|-------|-----------|
| STAFF / ADMIN | `staff_note` | STAFF can only update their own assigned sessions (`teach_by`) |
| CUSTOMER | `customer_note` | Can only update notes for contracts they `purchased_by` |

## 5. Expired Rules — Contract & History

### 5.1 Contract Expiry Rules

A contract can be automatically marked as `EXPIRED` under two conditions:

#### Rule C1: Pre-Active Expiry by Date
**Trigger:** A contract is in a pre-ACTIVE status (`NEWLY_CREATED`, `CUSTOMER_REVIEW`, `CUSTOMER_CONFIRMED`, `CUSTOMER_PAID`, `PT_CONFIRMED`) AND its `end_date` has passed.

**Where checked:**
- `/api/contract/getAll` — auto-expires in-memory, writes back to DB
- `/api/contract/updateStatus` — rejects any status change on already-expired contracts

**Behavior:** Contract status → `EXPIRED`, `updated_at` → `now`

#### Rule C2: Active Expiry by Credits (PT / REHAB only)
**Trigger:** A contract is `ACTIVE` AND its `end_date` has passed AND all credits have been used.

**Definition of "used credits":**
```typescript
const usedCredits = history.filter(h =>
  h.status === 'NEWLY_CREATED' || h.status === 'CHECKED_IN'
).length
```
A session counts as a "used credit" if it has status `NEWLY_CREATED` or `CHECKED_IN`.

**Note:** Sessions with status `CANCELED` or `EXPIRED` do **not** consume credits.

**Where checked:**
- `/api/contract/getAll` — auto-expires in-memory, writes back to DB
- `/api/contract/updateStatus` — rejects transitions on already-expired contracts

#### Rule C3: Manual Force-Expire
**ADMIN** can directly set a contract to `EXPIRED` via `updateStatus`.

#### Rule C4: Activating a Contract (`PT_CONFIRMED` → `ACTIVE`) — Date Adjustment
When a customer activates a contract (`PT_CONFIRMED` → `ACTIVE`), if the current time (`now`) differs from the stored `start_date`:
- `start_date` is reset to `now`
- `end_date` is recalculated: `now + (original_end_date - original_start_date)`

This preserves the original contract duration when activation is delayed.

### 5.2 History / Session Expiry Rules

#### Rule H1: NEWLY_CREATED Session Auto-Expiry
**Trigger:** A session has status `NEWLY_CREATED` AND its session end time has passed.

**Session end time calculation:**
```
sessionEndTime = session.date + (session.to * 60 * 1000)
// e.g., date=2026-04-01 00:00:00 (ms), to=570 minutes (9:30 AM)
// sessionEndTime = 1743465600000 + (570 * 60000) = 1743465600000 + 34200000
```

**Where checked:**
- `/api/history/getAll`
- `/api/history/getByContract`
- `/api/history/updateStatus` — rejects status changes on expired sessions

**Important:** Only `NEWLY_CREATED` sessions can auto-expire. Sessions already at `CHECKED_IN`, `CANCELED`, or `EXPIRED` are never re-expired.

#### Rule H2: Cannot Create Sessions on Expired Contracts
If `contract.end_date < now`, attempting to create a session returns:
```
{ error: 'Contract has expired' } // HTTP 400
```
The contract's status is also updated to `EXPIRED`.

#### Rule H3: Cannot Update Sessions on Expired Contracts
If a session's contract has expired (`end_date < now`), updating the session returns:
```
{ error: 'Contract has expired' } // HTTP 400
```

#### Rule H4: Expired Sessions Cannot Be Modified
Once a session's end time passes while in `NEWLY_CREATED` status, it is marked `EXPIRED`. Any subsequent `updateStatus` call returns:
```
{ error: 'Session has expired and cannot be updated' } // HTTP 400
```

### 5.3 Summary Table

| Entity | Expiry Trigger | Resulting Status | Automatic? |
|--------|---------------|-----------------|------------|
| Contract (pre-ACTIVE) | `end_date < now` | `EXPIRED` | ✅ |
| Contract (ACTIVE, PT/REHAB) | `end_date < now` AND credits exhausted | `EXPIRED` | ✅ |
| Contract (ACTIVE, PT_MONTHLY) | `end_date < now` | Remains `ACTIVE` (no credits to check) | ❌ |
| Session | Session end time `< now` AND status = `NEWLY_CREATED` | `EXPIRED` | ✅ |

## 6. Contract Workflow Lifecycle

```
[ADMIN/STAFF creates contract]
         │
         ▼
    NEWLY_CREATED
  (customer cannot see)
         │
    [ADMIN/STAFF sends to customer]
         │
         ▼
    CUSTOMER_REVIEW
         │
  [CUSTOMER confirms details]
         │
         ▼
  CUSTOMER_CONFIRMED
         │
  [CUSTOMER completes payment]
         │
         ▼
    CUSTOMER_PAID
         │
  [ADMIN/STAFF confirms receipt]
         │
         ▼
    PT_CONFIRMED
         │
  [CUSTOMER activates contract]
         │
         ▼
       ACTIVE
    (sessions can be booked)
         │
  ┌──────┴──────┐
  │             │
[Credits      [end_date
 exhausted]   passed]
  │             │
  ▼             ▼
 EXPIRED     EXPIRED
```

### Status Transitions by Role

```
ADMIN:  Can force-set any status to any other (except blocked by expiry rules)
STAFF:  NEWLY_CREATED → CUSTOMER_REVIEW
        CUSTOMER_PAID → PT_CONFIRMED
        NEWLY_CREATED/CUSTOMER_REVIEW/CUSTOMER_CONFIRMED → CANCELED
CUSTOMER: CUSTOMER_REVIEW → CUSTOMER_CONFIRMED
          CUSTOMER_CONFIRMED → CUSTOMER_PAID
          PT_CONFIRMED → ACTIVE
          CUSTOMER_REVIEW/CUSTOMER_CONFIRMED → CANCELED
```

## 7. History / Session Lifecycle

```
[User creates session]
         │
         ▼
    NEWLY_CREATED
   (pending check-in)
         │
    ┌────┴──────────┐
    │               │
[CUSTOMER         [STAFF/ADMIN
 checks in]       checks in]
    │               │
    │          (both checked in)
    │               │
    └──────►CHECKED_IN◄──────┘
              │
        [session time
         has passed]
              │
              ▼
          (no change,
           remains CHECKED_IN)

  CANCEL branch:
  NEWLY_CREATED → CANCELED (by any role, at any time before expiry)

  EXPIRY branch:
  NEWLY_CREATED → EXPIRED (when session end time < now)
```

### Dual Check-in Process

1. Customer or staff initiates check-in on a `NEWLY_CREATED` session
2. First check-in (by either party) sets the respective timestamp (`user_check_in_time` or `staff_check_in_time`)
3. Second check-in (by the other party) detects both timestamps present → status changes to `CHECKED_IN`
4. Each role can only check in once (idempotent)

## 8. Essential Information (Onboarding)

Every CUSTOMER must complete an onboarding questionnaire before their contract workflow begins. The form is defined in `src/lib/essentialInformation.ts`.

### Form Structure (34 questions across 6 sections)

| Section | Questions |
|---------|-----------|
| Thông tin cơ bản (Basic Info) | Full name, DOB, gender, email, phone, address, gym experience, current program |
| Thông số cơ thể (Body Metrics) | Height, weight, body fat %, muscle weight, chest/waist/hip/arm/thigh/calf measurements |
| Mục tiêu tập luyện (Training Goals) | Goals (checkbox), expectations |
| Sức khoẻ (Health) | Medical history, stimulants, injuries |
| Tập luyện (Workout) | Muscle focus, weekly days, preferred time, activity level, stress level, sleep quality, supplements |
| Ăn uống (Nutrition) | Daily meals, macros, eating habits |

### Essential Info Storage

- Stored as a **JSON string** in `user_setting.essential_information`
- `user_setting.essential_ready` = `true` when all required questions are answered and `is_submit=true`
- `is_submit: false` — saves progress without marking complete
- `is_submit: true` — validates all required questions answered before saving

### Required Questions
All questions except `FILE_UPLOAD` and `PAGE_BREAK` types are required.

## 9. Realtime Events

Powered by **Ably** (production only). Events are published after every mutation.

### Event Types

| Event Name | Trigger | Payload |
|------------|---------|---------|
| `contract.changed` | Contract create/update/delete/status-change | `{ entity_id, action, triggered_by, timestamp }` |
| `history.changed` | Session create/update/delete/status-change/note-update | `{ entity_id, action, triggered_by, timestamp }` |

### Actions

| Action | Applied To |
|--------|-----------|
| `create` | contract, history |
| `update` | contract, history |
| `delete` | contract, history (soft delete) |
| `update_status` | contract, history |
| `update_note` | history |

### Realtime Token

- Route: `GET /api/realtime/token`
- Returns Ably scoped token with `subscribe` permission on the user's own channel only
- Channel naming: `user:{userInstantId}`
- Only enabled in production (`NODE_ENV === 'production'`)
